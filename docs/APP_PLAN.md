## 1. 프로젝트 개요

### 1.1 앱 이름 (가칭)

**오목** — 간결한 이름, 검색·아이콘 식별 용이

### 1.2 목적

대한오목협회 급·단 체계를 반영한 난이도 구분, 렌주룰/일반룰 선택, 온라인 대전·관전·기보 리플레이를 제공하는 **무료 Android 오목 앱**을 만든다.

### 1.3 타깃 사용자

| 구분 | 설명 |
|------|------|
| 입문자 | 15~10급, 일반룰·렌주룰 학습, AI/엔진 대전 |
| 중급자 | 9~4급, 렌주룰 중심, 사람 대 사람 대전 |
| 상급·프로 | 3급~9단, 렌주룰 필수, 관전·기보 공유 |

### 1.4 핵심 가치

- **직관적 UI:** 스크롤 없이 한 화면에서 대부분의 조작 완료
- **공정한 대전:** 3급 이상은 렌주룰 강제
- **기록·공유:** Firebase 기보 저장, 리플레이, 관전
- **수익:** 배너 + 대국 종료 시 전면 광고 (무료 앱)

## 3. 기술 스택 및 아키텍처

### 3.1 권장 기술 스택

| 계층 | 기술 | 선정 이유 |
|------|------|-----------|
| **앱 셸** | React Native (Expo Dev Client) 또는 Capacitor + React | 기존 JS/TS 오목 엔진 재사용, Android 단일 타깃 |
| **UI** | React + TypeScript | 타입 안정성, 컴포넌트 재사용 |
| **상태** | Zustand 또는 Redux Toolkit | 대국·관전 실시간 상태 |
| **백엔드** | Firebase (Auth, Firestore, Realtime DB, Cloud Functions) | 실시간 대전·관전, 기보 저장 |
| **인증** | Firebase Auth + Google Play Games Services (선택) | Play 계정 1:1 게임 ID 연동 |
| **광고** | Google AdMob | 배너·전면 광고 표준 |
| **푸시** | Firebase Cloud Messaging (FCM) | 대국 초대·턴 알림 (2차) |

> **권장:** React Native + Expo + Firebase. JS 오목 엔진을 그대로 번들하고, AI는 네이티브/Web Worker 또는 Cloud Function으로 분리 가능.

## 4. 게임 모드 및 대전 구构

### 4.1 모드 목록

| 모드 | 설명 | 네트워크 |
|------|------|----------|
| **로컬 2인** | 한 기기에서 번갈아 착수 | 불필요 |
| **온라인 PvP** | 방 코드/초대로 원격 대전 | Firebase 실시간 |
| **vs 엔진 (JS)** | JavaScript 오목 엔진과 대국 | 로컬 (옵션: 기보만 업로드) |
| **vs AI** | AI 모듈과 대국 | 로컬 또는 Cloud Function |

### 4.2 대전 흐름 (온라인 PvP)

1. 호스트: 규칙·급/단·색(흑/백) 설정 → 방 생성
2. 게스트: 방 코드 입력 또는 초대 링크로 입장
3. 양측 준비 완료 → 대국 시작 (Firestore/RTDB 동기화)
4. 착수마다 서버 검증 (Cloud Function 또는 클라이언트+규칙 동기화)
5. 승패·무승부 확정 → 기보 저장 → 전면 광고 (설정 시)
6. 리플레이·재대국·로obby 복귀

### 4.3 컴퓨터 대전 흐름

1. **대전 유형 선택:** 엔진(JS) / AI
2. **난이도 선택:** 15급 ~ 9단 (급·단 표기)
3. **규칙:** 급수에 따라 일반/렌주 자동 또는 선택 (§5)
4. **색 선택:** 사용자 흑/백/랜덤
5. 대국 → 결과 → 기보 저장(선택) → 전면 광고

### 4.4 매칭 (선택 기능, 2차)

- 동일 급·단 ±1 허용 랜덤 매칭
- 1차 출시는 **방 코드 대전** 우선 (구현 단순)

## 5. 오목 규칙 (일반룰 / 렌주룰)

### 5.1 일반룰 (Freestyle)

- 15×15 또는 19×19 보드 (기본 **15×15**, 설정에서 19×19 선택 가능 — 1차는 15×15 고정 권장)
- 흑 선공, **5목 연속** 시 승리
- **3-3, 4-4, 장목(6목 이상)** 금지 없음
- **15급 ~ 4급** 구간에서 선택 가능

### 5.2 렌주룰 (Renju)

- 흑 선공
- **백:** 5목 연속 시 즉시 승리
- **흑:** 5목 연속 시 승리하나 **금수** 적용
  - **3-3** (열린 3이 동시에 2개)
  - **4-4** (열린 4가 동시에 2개)
  - **장목** (6목 이상 연속)
- 금수 착点 시 착수 불가 (UI: 해당 교차점 비활성·X 표시)
- **3급 이상:** 렌주룰 **강제** (일반룰 선택 UI 비활성 + ? 로 이유 설명)

### 5.3 규칙 적용 정책

| 사용자 설정 급·단 | 일반룰 | 렌주룰 |
|-------------------|--------|--------|
| 15급 ~ 4급 | ✅ 선택 가능 | ✅ 선택 가능 |
| 3급 ~ 1급 | ❌ | ✅ 필수 |
| 1단 ~ 9단 | ❌ | ✅ 필수 |

- PvP: **방장 설정** 기준, 양측 급·단 중 **높은 쪽** 기준으로 규칙 강제
- vs 컴퓨터: **선택한 난이도(급·단)** 기준

### 5.4 ? 도움말 (렌주룰)

- 짧은 그림/예시 3~4장 (3-3, 4-4, 장목)
- 「3급부터 렌주만」 한 줄 안내
- 장문 설명 금지 — 팝업 또는 바텀시트 1~2 스크린

## 6. 급·단 체계 및 AI/엔진 난이도 매핑

### 6.1 대한오목협회 기준 급·단

| 구분 | 범위 | 비고 |
|------|------|------|
| **급 (級)** | 15급 → 1급 | 숫자가 **작을수록** 상위 (15급=입문) |
| **단 (段)** | 1단 → 9단 | 아마추어 1급 다음이 1단, **9단=최고** (프로 동급) |

앱 내 표기: `15급`, `10급`, … `1급`, `1단`, … `9단`

### 6.2 난이도 슬라이더/그리드

- **한 줄 또는 2×12 그리드**로 급·단 선택 (스크롤 없이 — 작은 기기는 가로 스와이프 **한 줄**만 허용, 세로 스크롤 금지)
- vs 엔진 / vs AI 각각 동일 급·단 목록 제공

### 6.3 엔진(JS) 난이도 파라미터 (개념)

| 급·단 | 탐색 깊이 | 후보 수 | 특수 규칙 |
|-------|-----------|---------|-----------|
| 15~12급 | 1~2 | 많음, 랜덤성 | 실수 확률 높음 |
| 11~8급 | 2~4 | 중간 | 기본 패턴 |
| 7~4급 | 4~6 | VCF/VCT 일부 | 렌주 금수 준수 |
| 3급~1급 | 6~8 | VCF/VCT | 렌주 필수 |
| 1~3단 | 8~10 | 강한 휴리스틱 | — |
| 4~6단 | 10~12 | + 오프닝 북 | — |
| 7~9단 | 12+ / AI 위임 | 최대 | AI 모듈 권장 |

### 6.4 AI 난이도 (개념)

| 급·단 | 방식 |
|-------|------|
| 15~6급 | 가벼운 휴리스틱 + 약한 NN (온디바이스 또는 소형 모델) |
| 5급~3단 | 중형 모델 또는 강화 엔진 파라미터 |
| 4단~9단 | 전용 AI (더 깊은 탐색, NN, 또는 Cloud AI) |

- **9단**은 AI 전용 권장 (엔진만으로는 체감 난이도 부족 가능 → UI에 「AI 권장」 뱃지)

## 7. 오목 엔진 vs AI 구분

### 7.1 정의

| 구분 | JavaScript 오목 엔진 | AI |
|------|----------------------|-----|
| **본질** | 규칙·휴리스틱·알파베타/VCF 탐색 | 학습 모델·신경망·고급 평가 |
| **실행** | 앱 내 JS, 동기/Worker | ONNX/TFLite 또는 Cloud |
| **장점** | 가볍고 예측 가능, 오프라인 | 고단에서 인간다운 수 |
| **표시** | 「엔진 대전」 | 「AI 대전」 |

### 7.2 사용자 선택 UI

- 컴퓨터 대전 진입 시 **두 개의 큰 버튼**: `엔진` | `AI`
- 각 버튼 아래 한 줄: 「규칙 기반 · 가벼움」 / 「고급 · 고단 권장」
- 7단 이상 AI 선택 시 기본 하이라이트

### 7.3 기보 메타데이터

- `opponentType`: `human` | `engine` | `ai`
- `engineVersion` / `aiModelVersion` 저장 (리플레이·분석용)

## 8. 관전(스펙테이터) 기능

### 8.1 개요

- 진행 중인 대국에 **제3자**가 읽기 전용으로 접속
- 실시간 착수 반영 (지연 목표: **2초 이내**)

### 8.2 입장 방법

| 방법 | 설명 |
|------|------|
| **관전 코드** | 6~8자 영숫자 (대국 방과 별도 또는 동일 방 `spectate` 링크) |
| **공유 링크** | `omok://spectate/{roomId}` 또는 HTTPS 딥링크 |
| **기보 목록** | 「진행 중」 탭 (공개 방만, 2차) |

### 8.3 관전자 UI

- 오목판 + 현재 수순 + 흑/백 닉네임·급단
- **착수 불가** (터치 비활성)
- 채팅: 1차 **미제공** (욕설·스팸 리스크)
- 나가기 → 로obby (배너 유지)

### 8.4 동시 접속

- 관전자 **최대 50명/방** (Firestore 읽기 비용 고려, 조정 가능)
- RTDB `onValue` 또는 Firestore `onSnapshot` 으로 moves 배열 구독

### 8.5 프라이버시

- 방 설정: `public` | `private` (private는 코드 보유자만 관전)
- PvP 기본값: **private**

## 9. Firebase 데이터 설계

### 9.1 컬렉션 개요

| 컬렉션/경로 | 용도 |
|-------------|------|
| `users/{gameId}` | 프로필, 급·단, Play 계정 바인딩, 통계 |
| `rooms/{roomId}` | 진행 중 대국 상태 |
| `games/{gameId}` | 종료된 기보 (불변) |
| `replays/{gameId}` | 리플레이용 moves 압축 (games와 통합 가능) |

### 9.2 users/{gameId}

| 필드 | 타입 | 설명 |
|------|------|------|
| gameId | string | PK, 자동 생성 UUID v4 |

## 11. game 아이디 및 계정 정책

### 11.1 게임 ID 생성

- **최초 앱 실행** 시 `UUID v4` 생성 → `AsyncStorage`/SecureStore 로컬 저장
- Firebase `users/{gameId}` 문서 생성
- 닉네임 기본값: `Guest-` + gameId 앞 4자

### 11.2 Play 스토어 계정 1:1 정책

| 단계 | 동작 |
|------|------|
| 1 | Google Play Games Services 또는 Google Sign-In으로 Play 계정 식별자 획득 |
| 2 | `playAccountId` (SHA-256 해시)로 기존 `users` 조회 |
| 3 | **없음** → 현재 gameId를 해당 Play 계정에 **영구 바인딩** |
| 4 | **있음** → 로컬 gameId를 서버의 gameId로 **교체** (기기 변경 시 동일 ID 복원) |
| 5 | **재설치** 후 Play 로그인 → 동일 gameId 복구 |

### 11.3 제한

- **하나의 Play 계정 = 하나의 gameId** (Cloud Function에서 중복 생성 거부)
- Play 로그인 거부 시: 로컬 gameId만 사용 (기기 한정, 경고 1회)

### 11.4 닉네임·급단

- 닉네임 변경: 월 1회 제한 (스팸 방지)
- 급·단: **자가 신고** (실력 인증 없음) — ? 도움말에 명시
